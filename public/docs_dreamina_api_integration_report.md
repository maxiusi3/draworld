## 2. API 基础信息

### 2.1. 认证机制

即梦AI的API采用基于请求签名的认证方式，这是云服务商普遍采用的安全实践。每次API请求都必须包含正确的认证信息，否则将被拒绝。

- **认证凭证:** 
  - `AccessKeyID` (AK): 公开的访问密钥ID，用于标识用户。
  - `SecretAccessKey` (SK): 私有密钥，用于对请求进行签名，必须严格保密。
- **签名过程:** 
  1. 构造规范化的请求(Canonical Request)。
  2. 构造待签名的字符串(String to Sign)。
  3. 使用`SecretAccessKey`对待签名的字符串进行加密，生成签名(Signature)。
  4. 将签名信息附加到HTTP请求的Header中。
- **关键HTTP Header:**
  - `Host`: `visual.volcengineapi.com`
  - `Content-Type`: `application/json`
  - `X-Date`: 请求发起的UTC时间。
  - `X-Content-Sha256`: 请求体的SHA256哈希值。
  - `Authorization`: 包含签名信息的认证头。
  - `Region`: `cn-north-1` (固定值)
  - `Service`: `cv` (固定值，计算机视觉服务的缩写)

> **建议:** 强烈建议使用官方提供的SDK（如Python, Java, Go, NodeJS等）来处理认证和签名过程。SDK封装了复杂的签名算法，能极大简化开发工作，并降低因签名错误导致请求失败的风险。直接手动构造签名仅适用于没有可用SDK的场景。

### 2.2. 基础架构与调用方式

- **API Endpoint:** `https://visual.volcengineapi.com`
- **通信协议:** 所有API调用都必须通过HTTPS进行，以保证数据传输的安全性。
- **请求方法:** 主要使用`POST`方法提交数据。
- **数据格式:** 请求体和响应体均为`application/json`格式。
- **核心流程:** 即梦AI的视频生成是一个耗时操作，因此采用了**异步处理**模式。其调用流程分为两个主要步骤：
  1. **提交任务:** 客户端通过调用`CVSync2AsyncSubmitTask`接口发起一个视频生成请求。
  2. **查询结果:** 服务端立即返回一个`task_id`，客户端需使用此ID，通过轮询`CVSync2AsyncGetResult`接口来获取任务的最终结果。

## 3. Image2Video 核心接口分析

### 3.1. 提交任务 (CVSync2AsyncSubmitTask)

此接口用于启动一个“图生视频”任务。

**请求参数 (Query):**
- `Action`: 固定字符串 `CVSync2AsyncSubmitTask`
- `Version`: 固定字符串 `2022-08-31`

**请求参数 (Body):**
| 参数名 | 类型 | 是否必须 | 描述与要求 |
|---|---|---|---|
| `req_key` | string | 是 | 服务标识，对于图生视频功能，固定为 `jimeng_vgfm_i2v_l20`。 |
| `image_urls` | array of string | 二选一 | 图片的URL链接数组。**注意：** 这是推荐的使用方式，图片需能被公网访问。 |
| `binary_data_base64` | array of string | 二选一 | 图片的Base64编码字符串数组。`image_urls`优先于此参数。 |
| `prompt` | string | 否 | 描述期望视频内容的文本提示词。支持中英文，长度限制150字符。官方建议prompt内容应与输入图片相匹配，以获得更佳效果。 |
| `seed` | int | 否 | 随机种子，用于控制生成结果的确定性。默认为-1（随机）。范围：[-1, 2^64-1]。在调试或需要可复现结果时非常有用。 |
| `aspect_ratio` | string | 是 | 视频宽高比。必须从 `['16:9', '4:3', '1:1', '3:4', '9:16', '21:9', '9:21']` 中选择。**关键实践：** 应在前端或后端检测输入图片的尺寸，并选择最接近的宽高比，以避免视频内容被不当裁剪或拉伸。 |

**图片要求与限制:**
- **可访问性:** 若使用`image_urls`，必须确保链接是公网可访问的，否则火山引擎的服务无法抓取图片。
- **内容审核:** 输入图片会经过前置的内容安全审核。如果审核不通过，API将返回错误码 `50411` (Pre Img Risk Not Pass)，任务将直接失败。
- **格式:** 官方文档未明确限制图片格式，但标准的Web图片格式（如JPEG, PNG）应被支持。

**响应格式 (成功时):**
```json
{
    "code": 10000,
    "data": {
        "task_id": "12345678abcdefg"
    },
    "message": "Success",
    "request_id": "20240720..."
}
```
- `data.task_id`: 这是最重要的返回值，是后续查询任务状态和结果的唯一凭证。

### 3.2. 查询任务 (CVSync2AsyncGetResult)

此接口用于通过`task_id`获取异步任务的当前状态和最终结果。

**请求参数 (Query):**
- `Action`: 固定字符串 `CVSync2AsyncGetResult`
- `Version`: 固定字符串 `2022-08-31`

**请求参数 (Body):**
| 参数名 | 类型 | 是否必须 | 描述 |
|---|---|---|---|
| `req_key` | string | 是 | 服务标识，同提交任务接口，为 `jimeng_vgfm_i2v_l20`。 |
| `task_id` | string | 是 | 从`CVSync2AsyncSubmitTask`接口获取的任务ID。 |

**响应格式分析:**
- **任务处理中:**
```json
{
    "code": 10000,
    "data": {
        "status": "processing" // 或其他中间状态
    },
    ...
}
```
- **任务成功:**
```json
{
    "code": 10000,
    "data": {
        "status": "done",
        "video_url": "https://xxx.mp4",
        "resp_data": "{...}"
    },
    ...
}
```
  - `data.status`: 任务的最终状态，`done`表示成功。
  - `data.video_url`: **核心产出物**。生成的视频URL，**有效期为1小时**，视频时长固定为5秒。必须在有效期内及时下载或转存。
  - `data.resp_data`: 一个JSON字符串，包含了更详细的元数据，如视频md5、分辨率等。
- **任务失败:**
```json
{
    "code": 10000, // 注意：业务逻辑失败时，HTTP层面可能仍是成功的
    "data": {
        "status": "failed",
        "error": {
            "code": 50500, // 具体错误码
            "message": "Internal Error" // 错误信息
        }
    },
    ...
}
```
  - `data.status`: `failed`表示任务失败。
  - `data.error`: 包含具体的错误码和错误信息，是排查问题的重要依据。

## 4. 技术集成架构建议

基于即梦AI的异步特性，我们推荐采用前后端分离的架构，由后端负责与AI API的直接交互和轮询，前端负责用户交互和结果展示。

### 4.1. 后端集成方案 (Node.js/Python/Java等)

**核心职责:**
1.  **API密钥管理:** 安全地存储和管理`AccessKeyID`和`SecretAccessKey`。
2.  **封装API:** 提供内部接口供前端调用，隐藏火山引擎API的复杂性。
3.  **任务提交:** 接收前端的用户请求（图片、prompt等），调用`CVSync2AsyncSubmitTask`提交任务。
4.  **异步轮询:** 实现健壮的后台轮询逻辑，持续查询任务状态。
5.  **结果通知:** 任务完成后，通过适当方式（如WebSocket、Server-Sent Events或回调）通知前端，或允许前端通过一个结果接口查询。
6.  **视频转存:** (可选但推荐) 由于生成的`video_url`只有1小时有效期，建议后端在获取到URL后，立即将其下载并转存到自己的对象存储服务（如AWS S3, 阿里云OSS）中，生成一个永久有效的链接返回给前端。

**轮询机制设计:**
- **轮询策略:** 采用**指数退避**（Exponential Backoff）策略进行轮询，而非固定间隔轮询。
  - **初始间隔:** 从一个较短的间隔开始，例如2-3秒。
  - **增加间隔:** 如果任务仍在处理中，逐渐增加轮询间隔（例如每次乘以1.5或2），直到一个最大间隔（如15-20秒）。
  - **目的:** 这种方式在任务早期能快速获取结果，在任务后期则能显著减少对API的无效请求，降低服务端压力和潜在的请求频率限制风险。
- **超时处理:** 设置一个总的超时时间（例如5分钟）。如果超过这个时间任务仍未完成，应将其标记为失败，并停止轮询，记录日志以备排查。
- **数据库状态管理:** 后端应使用数据库（如MySQL, PostgreSQL, MongoDB）记录每个任务的状态。至少包含以下字段：`internal_task_id`, `user_id`, `dreamina_task_id`, `status` (`pending`, `processing`, `success`, `failed`), `source_image_url`, `final_video_url`, `error_message`。

### 4.2. 前端异步处理策略 (Vue.js/React等)

**核心职责:**
1.  **用户交互:** 提供上传图片、输入文本的界面。
2.  **发起请求:** 调用后端封装好的“创建视频任务”接口。
3.  **状态展示:** 在后端轮询期间，向用户清晰地展示任务状态，例如“视频生成中，请稍候...”、“排队中...”、“处理中...”。
4.  **结果接收与展示:** 
    - **轮询模式:** 前端可以定期请求后端的“任务状态”接口，获取最新状态。这是最简单的实现方式。
    - **推送模式 (更优):** 使用WebSocket或SSE。前端与后端建立长连接，一旦后端完成任务，立即将结果（最终视频URL）推送给前端，用户体验更佳，且能减少无效的HTTP请求。

### 4.3. 错误处理和重试机制

- **前端:** 
  - 友好地向用户展示错误信息，例如“图片内容不合规，请更换图片”、“服务暂时不可用，请稍后再试”。
  - 对于可重试的错误（如网络问题），可以提供一个“重试”按钮。
- **后端:**
  - **API请求层面:** 对于网络错误或可临时性的服务端错误（如HTTP 5xx），可以进行有限次数的重试（例如2-3次）。
  - **业务逻辑层面:** 仔细记录来自即梦AI的`error.code`和`error.message`，根据错误类型决定是否可以安全地自动重试。例如，对于`Internal Error`可以尝试重试一次，但对于`Pre Img Risk Not Pass`则不应重试，应直接将失败结果返回给用户。

### 4.4. 安全性考虑

- **API密钥管理:**
  - **严禁硬编码:** 绝对禁止在前端代码或代码仓库中硬编码`AccessKeyID`和`SecretAccessKey`。
  - **环境变量:** 在后端服务器上，通过环境变量来配置密钥。这是业界标准实践。
  - **密钥管理服务 (KMS):** 对于生产环境，推荐使用云服务商提供的KMS（如AWS KMS, Google Cloud KMS）来集中、安全地管理和轮换密钥，进一步提升安全性。
- **访问控制:** 后端接口需要有自己的用户认证和授权机制，确保只有登录用户才能调用视频生成功能，防止API被恶意滥用。
- **输入验证:** 在将用户输入（prompt文本、图片）发送给即梦AI之前，后端应进行自己的输入验证和清洗，防止潜在的注入攻击或格式错误。

## 5. 最佳实践与案例总结

### 5.1. 关键实践总结

1.  **优先使用官方SDK:** 简化认证流程，避免重复造轮子。
2.  **实施指数退避轮询:** 避免对API的过度请求，提高系统效率和健壮性。
3.  **后端处理异步任务:** 将复杂的轮询、状态管理和错误处理逻辑放在后端，前端只负责展示和简单交互。
4.  **视频资源及时转存:** 破解1小时URL有效期限制，保证业务数据的持久性。
5.  **动态匹配宽高比:** 前端或后端预处理图片，向API传入最合适的`aspect_ratio`，保证视频质量。
6.  **详细日志记录:** 记录完整的请求、响应和错误信息，特别`task_id`和`request_id`，是排查问题的生命线。

### 5.2. 潜在问题与解决方案

- **问题: 生成视频耗时过长，用户等待体验不佳。**
  - **解决方案:** 
    - **优化前端体验:** 使用更生动的加载动画和状态提示（如“正在渲染第一秒...”、“即将完成...”），让等待过程不那么枯燥。
    - **引入通知机制:** 对于Web应用，可以请求浏览器通知权限，当视频生成成功后，即使用户切换了标签页也能收到通知。
    - **邮件/短信通知:** 对于需要更长时间处理或允许离线的场景，可以在任务完成后通过邮件或短信通知用户回来查看。

- **问题: `video_url`过期导致视频无法访问。**
  - **解决方案:** 必须实现后端自动转存机制。一旦轮询到成功结果，立即触发一个后台作业将视频下载并上传到自己的持久化存储中。

- **问题: API成本失控。**
  - **解决方案:** 
    - **设置速率限制:** 在后端对用户调用频率进行限制（例如，每个用户每分钟只能创建1个任务）。
    - **增加预览/确认步骤:** 在正式提交任务前，可以增加一个确认步骤，让用户检查自己的输入，减少因误操作导致的无效调用。
    - **监控与告警:** 设置API调用量的监控，当调用次数或费用超过阈值时，自动发送告警。

## 6. 结论与后续步骤

即梦AI的Image2Video API提供了一个强大而直接的功能。其核心的技术挑战在于如何优雅地处理其异步工作流和管理API调用的生命周期。本报告提出的前后端分离、后端轮询与转存、前端状态同步的架构方案，能够有效地应对这些挑战。

**建议的下一步开发流程:**
1.  **后端开发:** 
    - 选择合适的技术栈（如Node.js + Express/Koa, Python + Flask/Django）。
    - 引入官方SDK，配置好密钥环境变量。
    - 开发任务提交接口和任务状态查询接口。
    - 实现数据库表结构和异步轮询逻辑。
    - （可选，但建议）集成对象存储服务，完成视频转存功能。
2.  **前端开发:**
    - 开发图片上传和Prompt输入界面。
    - 调用后端接口提交任务。
    - 实现对任务状态的轮询或通过WebSocket接收推送，并更新UI。
    - 在获取到最终视频URL后，进行播放展示。
